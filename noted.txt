4.2 ผลการทดสอบระบบ

การทดสอบระบบดำเนินการผ่าน API โดยใช้ชุดคำถามตัวอย่างจำนวน 50 กรณี ครอบคลุมคำถามทั่วไป ข้อมูลภาควิชา ข้อมูลหลักสูตรและรายวิชา ข้อมูลอาจารย์ ข้อมูลการติดต่อ และคำถามนอกขอบเขต การประเมินพิจารณาจากความสำเร็จของ API ความตรงประเด็นของคำตอบ การพบคำสำคัญที่คาดหวัง ความเหมาะสมของแหล่งอ้างอิง และเวลาในการตอบสนอง

4.2.1 เกณฑ์การประเมินผลการทดสอบระบบ

เกณฑ์ที่ใช้ในการประเมินผลประกอบด้วย 6 ด้าน ได้แก่

1. ความสำเร็จของ API: endpoint ตอบกลับด้วย HTTP 200 และ response มีโครงสร้างถูกต้อง
2. ความตรงประเด็นของคำตอบ: คำตอบเกี่ยวข้องกับคำถามและอยู่ในขอบเขตของระบบ
3. การพบคำสำคัญ: คำตอบมีข้อมูลหลักหรือ expected keywords ตามที่กำหนด
4. ความเหมาะสมของแหล่งอ้างอิง: แหล่งข้อมูลที่ใช้สอดคล้องกับประเภทคำถาม
5. เวลาในการตอบสนอง: วัดเวลาตั้งแต่ส่ง request จนได้รับ response
6. การจัดการคำถามนอกขอบเขต: ระบบสามารถปฏิเสธหรือชี้แจงข้อจำกัดได้เหมาะสม

ผลการประเมินแบ่งเป็น 3 ระดับ คือ ผ่าน ผ่านบางส่วน และไม่ผ่าน โดย "ผ่าน" หมายถึงคำตอบตรงคำถามและมีข้อมูลสำคัญครบถ้วน "ผ่านบางส่วน" หมายถึงคำตอบเกี่ยวข้องกับคำถามแต่ข้อมูลหรือแหล่งอ้างอิงยังไม่ครบถ้วน และ "ไม่ผ่าน" หมายถึงคำตอบผิดประเด็น ไม่พบข้อมูลสำคัญ หรือ API เกิดข้อผิดพลาด

ตารางที่ 4.2.1 ตัวอย่างเกณฑ์การตรวจผลคำตอบจากระบบ

4.2.2 ผลการทดสอบระบบตามหมวดคำถาม

ตารางที่ 4.2.2 ผลการทดสอบระบบด้วยชุดตัวอย่างคำถามในแต่ละรายการทดสอบ

จากตารางที่ 4.2.2 ระบบสามารถตอบคำถามได้ผ่านตามเกณฑ์ทุกหมวด โดยมีเวลาเฉลี่ยรวม 3.42 วินาที หมวดคำถามทั่วไปและคำถามนอกขอบเขตใช้เวลาน้อย เนื่องจากสามารถตอบกลับได้โดยตรง ส่วนหมวดข้อมูลหลักสูตรและรายวิชาใช้เวลาสูงที่สุด เนื่องจากต้องค้นคืนข้อมูลจากฐานความรู้และสร้างคำตอบด้วยแบบจำลองภาษา

4.2.3 ผลการตรวจสอบแหล่งอ้างอิงของคำตอบ

การตรวจสอบแหล่งอ้างอิงใช้เพื่อประเมินว่าระบบเลือกใช้แหล่งข้อมูลเหมาะสมกับประเภทคำถามหรือไม่ โดยข้อมูลภาควิชา หลักสูตร รายวิชา และอาจารย์ควรมาจากฐานความรู้ภายใน ส่วนคำถามทั่วไปและคำถามนอกขอบเขตสามารถตอบจากกฎการทำงานของระบบได้โดยไม่จำเป็นต้องค้นคืนเอกสาร

ตารางที่ 4.2.3 ผลการตรวจสอบแหล่งอ้างอิงของคำตอบในแต่ละหมวดคำถาม

หมวดคำถาม | แหล่งข้อมูลที่ใช้ | ผลการตรวจสอบ
ข้อมูลภาควิชา | Local RAG | ตรง
ข้อมูลหลักสูตร/รายวิชา | Local RAG | ตรง
ข้อมูลอาจารย์ | Local RAG | ตรง
ข้อมูลการติดต่อ | Local RAG / Web Search | ตรงบางส่วน
คำถามทั่วไป | Direct Response | ตรง
คำถามนอกขอบเขต | Direct Rejection | ตรง

จากตารางที่ 4.2.3 พบว่าระบบใช้ Local RAG ได้ตรงกับหมวดข้อมูลหลักของภาควิชา ได้แก่ ข้อมูลภาควิชา หลักสูตร รายวิชา และอาจารย์ ส่วนหมวดข้อมูลการติดต่อจัดเป็นตรงบางส่วน เนื่องจากบางกรณีมีการใช้ Web Search ร่วมกับฐานความรู้ภายใน

4.2.4 ขอบเขตและแนวทางการประเมินผลเชิงปริมาณ

จำนวนกรณีที่ API ตอบกลับสำเร็จ | 50 กรณี
อัตราการตอบกลับผ่าน API | 100%
เวลาเฉลี่ยในการตอบ | 3.42 วินาที
หมวดที่ตอบสนองเร็วที่สุด | คำถามนอกขอบเขต
หมวดที่ใช้เวลามากที่สุด | ข้อมูลหลักสูตร/รายวิชา

การทดสอบเชิงปริมาณครั้งนี้มีวัตถุประสงค์เพื่อตรวจสอบความสามารถของระบบภายใต้ชุดคำถามตัวอย่างที่ครอบคลุมหมวดการใช้งานหลัก ไม่ได้พิจารณาเฉพาะการมีคำตอบเท่านั้น แต่พิจารณาร่วมกับความสำเร็จของ API ความตรงประเด็นของคำตอบ การพบคำสำคัญที่คาดหวัง และความสอดคล้องของแหล่งอ้างอิง

อย่างไรก็ตาม จำนวนคำถามทดสอบถูกควบคุมให้อยู่ในระดับที่เหมาะสม เนื่องจากการเรียกใช้ LLM และ Web Search มีต้นทุนด้านค่าใช้จ่ายและเวลาในการประมวลผล ดังนั้น ผลการทดสอบจึงสะท้อนความสามารถของระบบในชุดคำถามตัวอย่าง มากกว่าการยืนยันว่าระบบสามารถตอบได้ถูกต้องในทุกสถานการณ์


ส่วนสำหรับแทนที่ตั้งแต่หัวข้อ 6 เป็นต้นไป

6. สรุป

โครงงานนี้เป็นการพัฒนาระบบต้นแบบแชตบอตผู้ช่วยอัจฉริยะสำหรับสนับสนุนการให้ข้อมูลของภาควิชาวิศวกรรมไฟฟ้า โดยประยุกต์ใช้ Retrieval-Augmented Generation ร่วมกับแบบจำลองภาษาขนาดใหญ่ ระบบสามารถค้นคืนข้อมูลจากเอกสารและเว็บไซต์ทางการของภาควิชาและคณะ แล้วนำมาสังเคราะห์เป็นคำตอบให้แก่ผู้ใช้ผ่านส่วนติดต่อผู้ใช้และ API ที่พัฒนาขึ้น

จากการทดสอบด้วยชุดคำถามจำนวน 50 คำถาม พบว่าระบบสามารถตอบกลับผ่าน API ได้ครบทุกกรณี มีเวลาเฉลี่ยในการตอบ 3.42 วินาที และสามารถตอบคำถามที่อยู่ในขอบเขตข้อมูลของภาควิชาได้ตามเกณฑ์ที่กำหนด ผลการทดสอบแสดงให้เห็นว่าแนวทาง RAG เหมาะสมกับงานให้ข้อมูลของภาควิชา เนื่องจากช่วยให้คำตอบอ้างอิงจากฐานความรู้ที่จัดเตรียมไว้ อย่างไรก็ตาม ระบบยังเป็นต้นแบบที่ทดสอบภายใต้ชุดคำถามตัวอย่างและข้อมูลสาธารณะที่มีอยู่ จึงควรมีการประเมินเพิ่มเติมกับผู้ใช้งานจริงก่อนนำไปใช้งานในสภาพแวดล้อมจริง

7. ข้อเสนอแนะ

ควรเพิ่มความครอบคลุมและความลึกของฐานความรู้ให้มากขึ้น เนื่องจากข้อมูลที่ใช้ในระบบปัจจุบันเป็นข้อมูลสาธารณะที่สามารถค้นหาได้จากเว็บไซต์หรือเอกสารทางการเป็นหลัก หากต้องการให้ระบบมีประโยชน์ต่อผู้ใช้งานทุกกลุ่ม เช่น นักศึกษา ผู้สนใจเข้าศึกษาต่อ อาจารย์ และเจ้าหน้าที่ ควรเพิ่มข้อมูลเชิงลึก เช่น ขั้นตอนการดำเนินงานภายใน คำถามที่พบบ่อย รายละเอียดแบบฟอร์ม แนวทางติดต่อหน่วยงาน และข้อมูลที่ช่วยลดภาระการตอบคำถามซ้ำของบุคลากร

หากต้องการพัฒนาระบบไปสู่การใช้งานจริงในระดับ production ควรต่อยอดจากระบบพื้นฐานที่มีอยู่แล้ว เช่น rate limiting, logging และ health check ให้ครอบคลุมการใช้งานจริงมากขึ้น โดยเพิ่มระบบจัดการฐานความรู้สำหรับผู้ดูแล ระบบประเมินคำตอบจากผู้ใช้ การบันทึกคำถามที่ระบบตอบไม่ได้ dashboard สำหรับติดตามคุณภาพคำตอบและค่าใช้จ่าย API รวมถึงการแจ้งเตือนเมื่อบริการสำคัญผิดปกติ นอกจากนี้ควรออกแบบแนวทางควบคุมค่าใช้จ่ายจากการเรียกใช้ LLM และ Web Search เช่น การทำ caching สำหรับคำถามที่พบบ่อย และการลดจำนวนครั้งในการเรียกใช้แบบจำลองภาษา เพื่อให้ระบบรองรับผู้ใช้งานหลายกลุ่มได้อย่างเหมาะสม
