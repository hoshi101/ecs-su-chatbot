## Section 4 — System Overview

> Presentation style tip: Use the system architecture diagram on the poster as your visual aid. Trace the flow from left to right: user → Streamlit → FastAPI → LangGraph → response. Keep this section high-level — the detailed node explanations come in Section 5.

> Time: 1 minute

**English:**

The system follows a client-server architecture. The user interacts with a Streamlit web interface, which sends requests to a FastAPI backend via REST API. The backend hands each query to the core intelligence of the system: a LangGraph state machine. This state machine orchestrates four specialized nodes — a router, a RAG retrieval engine, a web search engine, and a response generator. All document knowledge is stored in a Qdrant vector database hosted on the cloud. External search capability is provided by the Tavily API, restricted to official Finansia Hero domains.

The entire workflow is traced in real time, so users can see exactly how the chatbot decided to answer their question — which path it chose, which documents it found, and what reasoning it applied.

**ภาษาไทย:**

Workflow ทั้งหมดถูก trace แบบ real-time ผู้ใช้จึงเห็นได้ทันทีว่าแชทบอทเลือกเส้นทางใด ค้นพบเอกสารใด และใช้เหตุผลอะไรในการตอบ

## Section 5 — The Four-Node Workflow (Core Technical Section)

> Presentation style tip: This is the most important section. Point to the LangGraph diagram on the poster as you explain each node. Look at the diagram, then look at the evaluator, then continue. Slow down here — do not rush through the nodes. Pause between each node.

> Time: 3 minutes

**English:**

The heart of this system is a four-node LangGraph state machine. Each node has a distinct responsibility. Let me walk you through them.

**Node 1 — Router Node.**
Every query enters here first. The Router does two things. First, it performs query enhancement: the original user question is sent to the LLM, which rewrites it to be more specific and information-rich. For example, "how do I stop my loss?" becomes "how does the Stop-Loss order type work on the Finansia Hero platform and what are the configuration options?" This enhanced query produces far better search results. Second, the Router makes a routing decision: should this query go to the RAG system, the web search, receive a direct answer, or be flagged as out-of-scope? This decision is returned as a structured output from the LLM.

**Node 2 — RAG Lookup Node.**
If the Router decides RAG is appropriate, the enhanced query is converted into a vector using the BGE-M3 embedding model and compared against 1,737 or more document chunks stored in Qdrant using cosine similarity. The top-matching chunks are retrieved. Then — and this is the key innovation — the node applies an LLM-as-Judge mechanism: it sends the retrieved documents to the LLM with a specific prompt asking: "Are these documents sufficient to answer the user's question?" If the LLM judge says yes, the workflow moves to response generation. If the judge says no, the workflow escalates to web search automatically.

**Node 3 — Web Search Node.**
When RAG is insufficient — either because the documents do not contain the answer or the similarity scores fall below the threshold — this node performs a domain-restricted web search via the Tavily API. The search is constrained to official Finansia Hero domains, preventing irrelevant or potentially incorrect information from external sources from entering the response.

**Node 4 — Answer Generation Node.**
This final node takes either the retrieved document chunks or the web search results and generates a natural language response using Gemini 2.5 Flash with a temperature setting of 0.3. The low temperature setting ensures consistent, factually grounded answers appropriate for a financial domain where accuracy is paramount.

**ภาษาไทย:**

หัวใจของระบบคือ LangGraph State Machine 4 โหนด แต่ละโหนดมีหน้าที่เฉพาะ ขอเดินผ่านแต่ละโหนดให้ฟัง

**โหนด 1 — Router Node**
ทุกคำถามจะเข้ามาที่นี่ก่อน Router ทำ 2 สิ่ง อย่างแรกคือ Query Enhancement: คำถามเดิมของผู้ใช้จะถูกส่งให้ LLM ปรับให้เฉพาะเจาะจงและมีข้อมูลมากขึ้น เช่น "Stop-Loss ใช้ยังไง?" กลายเป็น "ประเภทคำสั่ง Stop-Loss บน Finansia Hero ทำงานอย่างไร และมีตัวเลือกในการตั้งค่าอะไรบ้าง?" คำถามที่ปรับแล้วค้นหาได้แม่นยำกว่ามาก อย่างที่สองคือการตัดสิน routing: คำถามนี้ควรไปที่ RAG, Web Search, ตอบโดยตรง หรืออยู่นอกขอบเขต? การตัดสินใจนี้ออกมาเป็น structured output จาก LLM

**โหนด 2 — RAG Lookup Node**
ถ้า Router ตัดสินว่าใช้ RAG คำถามที่ปรับแล้วจะถูกแปลงเป็น vector ด้วย BGE-M3 แล้วค้นหาในเอกสาร 1,737+ ชิ้นที่เก็บใน Qdrant ด้วย Cosine Similarity ชิ้นส่วนที่ใกล้เคียงที่สุดจะถูกดึงมา จากนั้นนี่คือนวัตกรรมสำคัญ โหนดนี้ใช้ LLM-as-Judge: ส่งเอกสารที่ค้นพบกลับไปให้ LLM พร้อม prompt ถามว่า "เอกสารเหล่านี้เพียงพอจะตอบคำถามผู้ใช้ไหม?" ถ้า LLM ตัดสินว่าพอ workflow ก็ไปสร้างคำตอบ ถ้าตัดสินว่าไม่พอ ก็จะ escalate ไป Web Search อัตโนมัติ

**โหนด 3 — Web Search Node**
เมื่อ RAG ไม่เพียงพอ ไม่ว่าจะเพราะเอกสารไม่มีคำตอบหรือ similarity score ต่ำกว่า threshold โหนดนี้จะค้นเว็บผ่าน Tavily API แบบจำกัดโดเมน เพื่อให้มั่นใจว่าข้อมูลที่ได้มาจากแหล่งทางการของ Finansia Hero เท่านั้น ไม่ใช่เว็บทั่วไปที่อาจไม่ถูกต้อง

**โหนด 4 — Answer Generation Node**
โหนดสุดท้ายนี้รับเอกสารที่ค้นพบหรือผลลัพธ์จาก Web Search แล้วสร้างคำตอบภาษาธรรมชาติด้วย Gemini 2.5 Flash โดยตั้ง temperature ไว้ที่ 0.3 ค่า temperature ต่ำนี้ทำให้คำตอบสม่ำเสมอและมีหลักฐานอ้างอิง เหมาะกับโดเมนการเงินที่ความถูกต้องสำคัญที่สุด

> Presenter Note:
> คำศัพท์ทางเทคนิคที่ต้องเข้าใจในส่วนนี้:
>
> "BGE-M3" คือ Embedding Model หมายความว่ามันแปลงข้อความให้เป็นตัวเลข (vector) ที่สื่อถึงความหมายของข้อความนั้น โมเดลนี้ทำงานได้หลายภาษาและฟรี ไม่ต้องจ่ายค่า API เหมือน OpenAI Embeddings
>
> "Cosine Similarity" คือวิธีวัดว่า vector สองตัวใกล้เคียงกันแค่ไหน ค่าใกล้ 1 หมายถึงความหมายคล้ายกันมาก ค่าใกล้ 0 หมายถึงไม่เกี่ยวกันเลย เหมือนการวัดมุมระหว่างสองทิศทาง ถ้าทิศทางเดียวกันแสดงว่าความหมายเหมือนกัน
>
> "LLM-as-Judge" คือการใช้ AI ประเมิน AI อีกตัว แทนที่จะให้คนมานั่งตรวจว่าเอกสารที่ค้นมาเพียงพอไหม เราให้ LLM ตัดสินเองก่อน ถ้าไม่พอก็ส่งต่อให้ค้นเว็บ ซึ่งทำให้ระบบทำงานได้อัตโนมัติโดยไม่ต้องมีคนคอยดู
>
> "Query Enhancement" คือการให้ AI ปรับปรุงคำถามก่อนค้นหา เหมือนนักสืบที่ขอข้อมูลเพิ่มเติมก่อนออกไปค้นหาหลักฐาน
>
> "Temperature 0.3" ค่า temperature ควบคุมความสร้างสรรค์ของ AI ค่าต่ำ (เช่น 0) คือตอบแบบ conservative ค่าสูง (เช่น 1) คือตอบแบบสร้างสรรค์ เราตั้ง 0.3 เพราะต้องการความถูกต้อง ไม่ใช่ความแปลกใหม่ ในโดเมนการเงิน ถ้า AI แต่งข้อมูลเพราะ temperature สูงเกินไป อาจทำให้คนไปตัดสินใจซื้อขายผิดพลาดได้
>
> "Similarity Threshold" คือเกณฑ์ที่กำหนดว่า cosine similarity เท่าไรถึงถือว่า "เกี่ยวข้อง" ถ้าตั้ง threshold สูง (เช่น 0.9) คือจะดึงเฉพาะเอกสารที่เกี่ยวข้องมากเท่านั้น แต่อาจพลาดเอกสารที่มีประโยชน์ ถ้าตั้งต่ำ (เช่น 0.5) ดึงเยอะแต่อาจมีเอกสารที่ไม่เกี่ยวมาด้วย

## Section 6 — Document Processing Pipeline

> Presentation style tip: Point to the pipeline diagram on the poster. This section can be kept brief unless an evaluator asks for more detail. Have the numbers ready: 1,737+ chunks, 1,024 dimensions.

> Time: 1 minute

**English:**

Before the chatbot can answer anything, we must process the raw documents into a searchable knowledge base. The pipeline has four steps.

Step one is ingestion: we load all source documents using format-specific loaders — PyPDFLoader for PDF manuals, CSVLoader for CSV data files, and JSONLoader for JSON configuration data. This covers 59 megabytes of heterogeneous financial documentation.

Step two is chunking: each document is split into smaller pieces using a RecursiveCharacterTextSplitter with a chunk size of 1,000 characters and an overlap of 200 characters. The overlap ensures that contextual information at chunk boundaries is not lost.

Step three is embedding: each chunk is converted into a 1,024-dimensional vector using the BGE-M3 multilingual embedding model. This model handles Thai and English text natively, producing semantically meaningful representations for both languages.

Step four is storage: all vectors and their associated metadata are uploaded to Qdrant Cloud, where they are indexed for cosine similarity search. The result is a knowledge base of 1,737 or more searchable chunks.

**ภาษาไทย:**

ก่อนที่แชทบอทจะตอบคำถามได้ เราต้องประมวลผลเอกสารดิบให้เป็นฐานความรู้ที่ค้นหาได้ pipeline มี 4 ขั้นตอน

ขั้นที่สองคือ chunking: แบ่งเอกสารแต่ละชิ้นออกเป็นส่วนย่อยๆ ด้วย RecursiveCharacterTextSplitter ขนาด 1,000 ตัวอักษร และ overlap 200 ตัวอักษร overlap นี้ช่วยให้บริบทที่อยู่รอยต่อระหว่าง chunk ไม่สูญหาย

ขั้นที่สามคือ embedding: แปลงแต่ละ chunk เป็น vector 1,024 มิติด้วย BGE-M3 ซึ่งรองรับทั้งภาษาไทยและอังกฤษโดยธรรมชาติ ให้ representation ที่สื่อถึงความหมายได้ดีทั้งสองภาษา

ขั้นที่สี่คือ storage: อัปโหลด vector และ metadata ทั้งหมดขึ้น Qdrant Cloud เพื่อ index สำหรับการค้นหาด้วย Cosine Similarity ผลลัพธ์คือฐานความรู้ที่มีชิ้นส่วนที่ค้นหาได้มากกว่า 1,737 ชิ้น

> Presenter Note:

### Q1 — Why use RAG instead of fine-tuning the LLM directly?

**English:**
Fine-tuning requires large labeled datasets, significant computational cost, and must be repeated every time the documentation changes. RAG allows us to update the knowledge base instantly by adding new documents to Qdrant — no retraining required. More importantly, RAG grounds responses in actual documentation, which directly reduces hallucination. In a financial domain where incorrect information can lead to monetary loss, this grounding is not optional — it is essential.

**ภาษาไทย:**
Fine-tuning ต้องใช้ข้อมูลที่ label แล้วจำนวนมาก มีค่าใช้จ่ายสูง และต้องทำใหม่ทุกครั้งที่เอกสารเปลี่ยน RAG อัปเดตฐานความรู้ได้ทันทีโดยเพิ่มเอกสารใหม่เข้า Qdrant ไม่ต้อง retrain เลย ที่สำคัญกว่า RAG บังคับให้คำตอบอ้างอิงจากเอกสารจริง ลดปัญหา Hallucination ได้โดยตรง ในโดเมนการเงินที่ข้อมูลผิดอาจทำให้เสียเงิน สิ่งนี้ไม่ใช่ตัวเลือก แต่จำเป็นอย่างยิ่ง

> Presenter Note: Hallucination คือ AI แต่งข้อมูลที่ไม่มีอยู่จริง ลองนึกภาพ AI บอกว่า "คุณสามารถซื้อขายได้ตลอด 24 ชั่วโมง" ทั้งที่จริงตลาดหุ้นปิดตอนเย็น — นักลงทุนเชื่อแล้วก็พลาดโอกาสหรือแย่กว่านั้น

### Q3 — Why use BGE-M3 instead of OpenAI Embeddings?

**English:**
BGE-M3 is open-source, so there are no per-call API costs for embedding. It ranks at the top of the MTEB multilingual benchmark for Thai-English mixed text, producing 1,024-dimensional vectors that capture semantic nuance better than lower-dimensional alternatives. For a system handling bilingual financial documentation, BGE-M3 is both more capable and more cost-effective than proprietary embedding APIs.

**ภาษาไทย:**
BGE-M3 เป็น Open-Source ไม่มีค่า API สำหรับ embedding รองรับข้อความผสมไทย-อังกฤษได้ดีที่สุดในเบนช์มาร์ค MTEB multilingual ให้ vector 1,024 มิติที่จับความหมายได้ละเอียดกว่า alternative ที่มิติต่ำกว่า สำหรับระบบที่จัดการเอกสารการเงินสองภาษา BGE-M3 ทั้งมีความสามารถสูงกว่าและคุ้มค่ากว่า embedding API แบบ proprietary

### Q4 — What happens when the documentation is updated?

**English:**
The document processing pipeline is fully independent of the chatbot runtime. When documentation changes, we re-run the pipeline on the updated or new documents — ingest, chunk, embed, and upsert into Qdrant. The chatbot immediately benefits from the updated knowledge base on the next query. No code changes, no model retraining, no downtime beyond the pipeline run itself.

**ภาษาไทย:**
Document Processing Pipeline แยกออกจาก chatbot runtime โดยสมบูรณ์ เมื่อเอกสารเปลี่ยน เราแค่รัน pipeline ใหม่กับเอกสารที่อัปเดต — ingest, chunk, embed และ upsert เข้า Qdrant แชทบอทได้รับประโยชน์จากฐานความรู้ที่อัปเดตทันทีในคำถามถัดไป ไม่ต้องแก้โค้ด ไม่ต้อง retrain ไม่มี downtime นอกจากเวลารัน pipeline

Workflow tracing serves two audiences. For administrators and evaluators, it provides auditability: every decision point is logged, which is critical for financial applications subject to regulatory scrutiny. ### Q7 — What does Temperature 0.3 mean and why not use a different value?

**English:**
Temperature controls the randomness of the language model's output. A temperature of zero produces the most deterministic response — the model always picks the highest-probability next token. A temperature of one introduces significant variation. We selected 0.3 because it produces consistent, factually grounded responses with minimal creative deviation. In a financial domain, a response that is 90% accurate but creatively phrased is less useful than one that is consistently accurate. High temperature in financial applications risks generating plausible-sounding but incorrect information — precisely the hallucination problem RAG is designed to mitigate.

**ภาษาไทย:**
Temperature ควบคุมความสุ่มของ output จาก language model ค่าศูนย์ให้คำตอบที่ deterministic ที่สุด โมเดลเลือก token ที่มี probability สูงสุดเสมอ ค่าหนึ่งทำให้มีความหลากหลายมาก เราเลือก 0.3 เพราะให้คำตอบที่สม่ำเสมอและอ้างอิงข้อเท็จจริงโดยไม่มีการเบี่ยงเบนเชิงสร้างสรรค์ ในโดเมนการเงิน คำตอบที่ถูกต้องสม่ำเสมอดีกว่าคำตอบที่ฟังดูสร้างสรรค์แต่ผิด Temperature สูงในแอปการเงินเสี่ยงสร้างข้อมูลที่ฟังดูน่าเชื่อแต่ผิดจริง ซึ่งเป็นปัญหา Hallucination ที่ RAG ออกแบบมาเพื่อลด

### Q8 — How does Query Enhancement work exactly?

**English:**
The original user query is sent to the LLM with a prompt that instructs it to rewrite the question to be more specific, expand abbreviations, add platform context, and use terminology likely to appear in official documentation. The enhanced query is then used for both the routing decision and the semantic search in Qdrant. For example, a user asking "what is SL?" might have their query enhanced to "how does the Stop-Loss order type function on the Finansia Hero platform and what are the risk management configurations available?" The enhanced form produces significantly higher cosine similarity scores when matched against documentation chunks.

**ภาษาไทย:**
คำถามเดิมของผู้ใช้จะถูกส่งให้ LLM พร้อม prompt ที่สั่งให้เขียนคำถามใหม่ให้เฉพาะเจาะจงมากขึ้น ขยายคำย่อ เพิ่มบริบทของแพลตฟอร์ม และใช้คำศัพท์ที่น่าจะปรากฏในเอกสารทางการ คำถามที่ปรับแล้วจะใช้ทั้งสำหรับการตัดสิน routing และการค้นหา semantic ใน Qdrant ตัวอย่างเช่น ผู้ใช้ถาม "SL คืออะไร?" อาจถูกปรับเป็น "ประเภทคำสั่ง Stop-Loss ทำงานอย่างไรบน Finansia Hero และมีการตั้งค่า risk management อะไรบ้าง?" รูปแบบที่ปรับแล้วให้ cosine similarity score สูงกว่าอย่างมีนัยสำคัญเมื่อ match กับ document chunks

### Q9 — How does the system handle out-of-domain queries?

**English:**
The Router Node evaluates whether a query is related to the Finansia Hero platform. If the LLM determines the query falls outside the defined scope — for example, a user asking about cryptocurrency prices or general investment advice not related to platform functionality — the system returns a polite message indicating the question is outside scope, without attempting to answer from insufficient or irrelevant information. This prevents the system from providing potentially harmful financial guidance it was not designed to give.

**ภาษาไทย:**
Router Node ประเมินว่าคำถามเกี่ยวข้องกับ Finansia Hero หรือไม่ ถ้า LLM ตัดสินว่าคำถามอยู่นอกขอบเขต เช่น ผู้ใช้ถามเรื่องราคา cryptocurrency หรือคำแนะนำการลงทุนทั่วไปที่ไม่เกี่ยวกับฟังก์ชันของแพลตฟอร์ม ระบบจะตอบอย่างสุภาพว่าคำถามอยู่นอกขอบเขต โดยไม่พยายามตอบจากข้อมูลที่ไม่เพียงพอหรือไม่เกี่ยวข้อง ซึ่งป้องกันไม่ให้ระบบให้คำแนะนำการเงินที่อาจเป็นอันตรายและไม่ได้ออกแบบมาเพื่อสิ่งนั้น

### Q10 — Can this system be adapted to other domains?

**English:**
Yes, and this is one of the core architectural decisions. The LangGraph state machine, Qdrant vector database, and LLM are all configurable components. To adapt the system to another domain — such as banking, insurance, or e-commerce — the process involves three changes: replace the document knowledge base with domain-appropriate documents, update the system prompt to reflect the new domain's terminology and scope, and adjust the domain-restricted web search to the relevant URLs. The routing logic, retrieval mechanism, LLM-as-Judge, and response generation nodes remain unchanged. This modularity was a deliberate design choice.

**ภาษาไทย:**
ได้ และนี่คือหนึ่งในการตัดสินใจออกแบบ architecture หลัก LangGraph State Machine, Qdrant Vector Database และ LLM ล้วนเป็น component ที่ปรับเปลี่ยนได้ การปรับระบบไปยังโดเมนอื่น เช่น ธนาคาร ประกันภัย หรือ e-commerce ต้องเปลี่ยน 3 อย่าง ได้แก่ เปลี่ยนฐานความรู้เอกสารเป็นเอกสารของโดเมนใหม่ อัปเดต system prompt ให้สะท้อนคำศัพท์และขอบเขตของโดเมนใหม่ และปรับ domain-restricted web search ให้ตรงกับ URL ที่เกี่ยวข้อง logic การ routing, retrieval mechanism, LLM-as-Judge และ response generation node ไม่จำเป็นต้องเปลี่ยน ความ Modular นี้เป็นการตัดสินใจออกแบบโดยเจตนา

## Section 12 — Quick Reference: Technical Term Glossary

|---|---|---|
| RAG (Retrieval-Augmented Generation) | Technique that retrieves relevant documents and feeds them to an LLM before generating a response | ดึงเอกสารที่เกี่ยวข้องมาให้ AI อ่านก่อน แล้วค่อยสร้างคำตอบ เหมือนเปิดหนังสือก่อนตอบ |
| LangGraph State Machine | A programmable flowchart for AI workflows where each step is a node and edges are conditional transitions | flowchart ที่รันได้สำหรับ AI แต่ละโหนดคือขั้นตอน แต่ละเส้นคือเงื่อนไขการเดินต่อ |
